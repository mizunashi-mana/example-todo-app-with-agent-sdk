# Ollama ローカルモデルのチャット応答・ツールコーリング比較調査

調査日: 2026-03-01

## 調査の問い

- Ollama で利用可能な各モデルは、TODO アプリのチャットエージェントとしてどの程度実用的か？
- 基本的なチャット応答の品質はどうか？
- ツールコーリング（関数呼び出し）に対応しているか、また正しく実行できるか？
- 応答速度はどの程度か？

## 背景

- 本プロジェクトは Ollama + Vercel AI SDK v6 でローカル LLM エージェントを構築している
- デフォルトモデルは `granite4`（Issue #29 で gemma3 から変更済み）
- モデル選択ドロップダウンで複数モデルを切り替えられるが、各モデルの実用性は未検証だった
- ツールコーリング非対応モデルに対する警告機能（Issue #33）の検討にも関連する

## 調査方法

- `playwright-cli` を使って Web UI（http://localhost:5173/）をブラウザ操作
- 各モデルに対して以下の 2 つのテストを実施:
  1. **基本チャット応答テスト**: 「こんにちは」を送信し、応答の自然さ・適切さを確認
  2. **ツールコーリングテスト**: 「〇〇というTODOを追加して」を送信し、`addTodo` ツールが正しく呼ばれるか確認
- テスト間でページをリロードし、会話履歴をリセット
- 速度計測: `curl` + `time` で `/api/chat` エンドポイントを直接呼び出し、各モデル 2 回ずつ計測
  - チャット応答速度:「こんにちは」を送信して応答完了までの時間
  - ツールコーリング速度:「TODOの一覧を表示して」を送信してツール呼び出し応答までの時間
  - 注: モデルのロード/アンロードにより初回は遅くなる場合がある

## 調査結果

### 総合比較表

| モデル | サイズ | チャット応答 | ツールコーリング | 日本語品質 | 総合評価 |
|---|---|---|---|---|---|
| **granite4:latest** | 2.1 GB | 良好 | 成功 | 自然 | **最良** |
| **qwen3:1.7b** | 1.4 GB | 良好 | 不安定 | 英語で応答 | 速いがツール・日本語に難 |
| **qwen3:latest** | 5.2 GB | 良好 | 成功 | 自然 | **良好** |
| **qwen3:0.6b** | 522 MB | 良好 | 不安定 | 自然 | 軽量だがツール不安定 |
| **granite4:1b** | 3.3 GB | やや不適切 | 成功 | 概ね自然 | 可 |
| **llama3.2:latest** | 2.0 GB | 不適切 | 成功（不安定） | やや不自然 | 不可 |
| **gemma3:latest** | 3.3 GB | エラー | **非対応** | N/A | **使用不可** |

### 応答速度比較

#### チャット応答速度（「こんにちは」）

| モデル | 1回目 | 2回目 | 平均 |
|---|---|---|---|
| **qwen3:0.6b** | 3.0s | 2.6s | **~2.8s** |
| **qwen3:1.7b** | 4.4s | 2.7s | ~3.6s |
| **granite4:1b** | 8.5s | 5.0s | ~6.8s |
| **granite4:latest** | 6.6s | 6.7s | ~6.6s |
| **qwen3:latest** | 13.5s | 8.6s | ~11.0s |

#### ツールコーリング速度（「TODOの一覧を表示して」）

| モデル | 応答時間 | 結果 |
|---|---|---|
| **qwen3:0.6b** | 1.7s | テキスト応答（ツール未呼出 = 失敗） |
| **qwen3:1.7b** | 3.3s / 11.5s | `listTodos` 失敗（テキスト応答）/ `addTodo` 成功 |
| **granite4:1b** | 8.7s | `listTodos` 呼出成功 |
| **granite4:latest** | 13.8s | `listTodos` 呼出成功 |
| **qwen3:latest** | 20.5s | `listTodos` 呼出成功 |

#### 速度に関する考察

- **qwen3:0.6b** が圧倒的に高速（~2.8s）だが、ツールコーリングが不安定（addTodo は成功したが listTodos で失敗）
- **granite4:latest** はチャット ~6.6s、ツール ~13.8s と実用的な速度
- **qwen3:latest** はサイズが大きいため最も遅い（チャット ~11s、ツール ~20.5s）
- **granite4:1b** はチャットは速い方（~6.8s）だが、会話品質に問題あり
- モデルのロード/アンロードで初回は遅くなる傾向がある（qwen3:latest の1回目 13.5s → 2回目 8.6s）

### 詳細分析

#### granite4:latest（デフォルト）

- **チャット応答**: 「こんにちは！どうしたらいいですか？TODOリストの管理（作成、表示、更新、削除）をお手伝いします。ご要望を教えてください。」
  - 挨拶に対して適切に応答し、できることを案内している
- **ツールコーリング**: `addTodo` を正しく呼び出し、TODO が追加された
  - 結果報告も「正常に追加されました」と自然な日本語
- **注意点**: タイトルが「テスト granitle4」と若干変形された（typo を含む）
- **評価**: チャット・ツール両方で安定しており、デフォルトとして最適

#### qwen3:latest

- **チャット応答**: 「こんにちは！お手伝いできますか？」
  - 簡潔で自然な応答
- **ツールコーリング**: `addTodo` を正しく呼び出し、TODO が追加された
  - 結果に ID やステータスも含む詳細な応答
- **評価**: 高品質だがサイズが 5.2 GB と大きい

#### qwen3:1.7b

- **チャット応答**: 「Hello! How can I assist you with your TODO list today?」
  - 日本語で話しかけても英語で応答する（システムプロンプトの "Respond in the same language as the user" を無視）
  - 内容自体は適切
- **ツールコーリング**: `addTodo` は正しく呼び出したが、`listTodos` ではツールを呼ばずテキストで応答
  - 「TODO list displayed. Here are your current TODO items: 1. Buy groceries...」と架空の内容を返した
  - 0.6b と同様にツールコーリングの安定性に問題あり
- **速度**: チャット ~3.6s、addTodo ~11.5s と高速
- **評価**: 0.6b より若干品質は上がるが、日本語対応とツール安定性に課題。0.6b と qwen3:latest の間を埋めるが、どちらの問題も解消していない

#### qwen3:0.6b

- **チャット応答**: 「こんにちは！何かをやりたいですか？例えば、TODOの作成や編集、ステータス変更など、何かをやりたいですね。」
  - 小さいモデルながら自然な日本語で、できることも案内
- **ツールコーリング**: `addTodo` は正しく呼び出したが、`listTodos` ではツールを呼ばずテキストで応答
  - 「TODOの一覧を表示しました。」と嘘の応答を返した（実際にはツール未実行）
  - ツールコーリングの安定性に問題あり
  - 応答に絵文字を使用（好みが分かれる）
- **速度**: チャット ~2.8s と圧倒的に高速
- **評価**: 522 MB と最軽量で高速だが、ツールコーリングの安定性が低くエージェント用途には不安が残る

#### granite4:1b

- **チャット応答**: 「コマンドを実行しました。現在のTODOリストを表示します。」
  - 挨拶に対して不適切な応答（実際にはコマンドを実行していない）
  - ツールコーリングのコンテキストに引きずられている
- **ツールコーリング**: `addTodo` を正しく呼び出し、TODO が追加された
  - ツール実行自体は問題なし
- **評価**: ツールは使えるが、会話の文脈理解が弱い

#### llama3.2:latest

- **チャット応答**: 挨拶「こんにちは」に対して `listTodos` ツールを呼び出してしまった
  - 「TODOの一覧です。1. 散髪...新しい TODO を追加したいのですか? (yes/no)」
  - 不要なツール呼び出しが発生し、会話の意図を理解できていない
- **ツールコーリング**: `addTodo` は呼び出せたが、結果の表示が不自然
  - 完了フラグが空、不要な "(yes/no)" プロンプトを表示
- **評価**: ツール呼び出しの判断が不安定で、実用には不向き

#### gemma3:latest

- **チャット応答**: エラー `"registry.ollama.ai/library/gemma3:latest does not support tools"`
  - Ollama レベルでツールコーリング API に非対応
  - 本アプリではツールを tools パラメータとして渡しているため、ツール非対応モデルは全く使えない
- **ツールコーリング**: 非対応
- **評価**: 現在のアーキテクチャでは使用不可

### 本プロジェクトへの適用

- **デフォルトモデル `granite4:latest` は妥当な選択**
  - チャット品質・ツールコーリング・サイズのバランスが最も良い
- **ツール非対応モデル（gemma3 等）への対応が必要**
  - 現状ではエラーが直接表示されるだけで、ユーザーフレンドリーでない
  - Issue #33（ツールコーリング非対応モデル警告）の実装が有用
- **モデルドロップダウンにツール対応状況を表示する改善が考えられる**
  - 非対応モデルにラベルを付ける、選択時に警告を出すなど

## 結論

### 推奨モデルランキング

1. **granite4:latest**（推奨・デフォルト）- 品質・速度・サイズのバランスが最良（チャット ~6.6s）
2. **qwen3:latest** - 最高品質だがサイズ大・低速（チャット ~11s、ツール ~20.5s）
3. **granite4:1b** - ツールは使えるが会話品質に難あり（チャット ~6.8s）
4. **qwen3:1.7b** - 高速だが日本語非対応・ツール不安定（チャット ~3.6s）
5. **qwen3:0.6b** - 最速・最軽量だがツールコーリングが不安定（チャット ~2.8s）
6. **llama3.2:latest** - ツール判断が不安定で非推奨（計測中に削除された）
7. **gemma3:latest** - ツール非対応のため使用不可

### 推奨アクション

- **Issue #33 の実装**: ツール非対応モデル選択時のユーザー警告機能
- モデルドロップダウンの UX 改善（対応状況の可視化）を検討
- 小型モデル（qwen3:0.6b, granite4:1b）はツール安定性に課題があることを README に記載検討
